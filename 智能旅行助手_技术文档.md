# 智能旅行助手项目技术文档

## 1. 项目概述

### 1.1 背景
本项目是“智能旅行助手”，主要完成旅行规划、天气查询、预算计算、地图展示和行程导出等能力，并且具备用户体系、个人历史方案持久化、异步任务处理、半结构化数据存储和流水线式 Agent 编排的全栈 Web 应用。

### 1.2 建设目标
1. 支持用户注册、登录与基础身份认证。
2. 支持用户保存并查询自己历史创建的旅行方案，方案与版本需持久化。
3. 支持基于历史方案继续生成新版本，保留版本链。
4. 采用流水线式多阶段 Agent 编排，降低系统复杂度并增强可控性。
5. 在行程生成与查看时引入天气预警提示，必要时对方案进行风险约束。
6. 支持异步生成与查询，避免长时间请求阻塞前端。
7. 提供 React 前端页面与 FastAPI 后端接口，实现前后端分离。

### 1.3 项目定位
本项目定位为“课程项目级、可演示、可扩展”的 AI 应用系统。重点不在实时协同，而在于构建一套结构清晰、可落地、可解释、可维护的旅行规划系统。系统保留 AI 的核心亮点，但以工程可控性为优先。

## 2. 需求分析

### 2.1 功能需求

#### 2.1.1 用户系统
- 用户注册：用户填写基础信息并创建账户。
- 用户登录：使用 JWT 完成身份认证。
- 用户信息管理：支持查看和修改个人基础资料。

#### 2.1.2 历史方案管理
- 用户可以保存自己创建的旅行方案。
- 用户可以查询自己历史生成过的方案列表。
- 用户可以查看某个方案的版本历史。
- 用户可以基于历史版本继续生成新的方案版本。
- 所有版本均持久化保存，不覆盖旧版本。

#### 2.1.3 用户画像
- 用户注册时填写基础偏好信息。
- 系统根据用户输入、历史方案和编辑行为形成用户画像。
- 用户画像用于后续行程生成、天气敏感提示和偏好推荐。

#### 2.1.4 智能行程规划
- 用户输入旅行城市、日期、预算、出行方式、住宿偏好等信息。
- 系统通过流水线式 Agent 生成完整旅行方案。
- 方案包含景点、酒店、餐饮、天气、预算、建议与每日安排。

#### 2.1.5 天气预警提示
- 用户打开某个方案时，系统检查旅行时间段内的天气信息与预警信息。
- 若存在暴雨、高温、大风、雷暴等影响出行的风险，系统向用户展示提示。
- 系统可以结合天气风险对方案生成结果进行约束，但不强制自动改写用户已保存方案。

#### 2.1.6 导出与展示
- 支持在前端展示地图、行程卡片、预算统计与天气信息。
- 支持导出为 PDF 。

### 2.2 非功能需求
- **可维护性**：模块边界清晰，Agent、API、数据库、前端分层明确。
- **可扩展性**：后续可增加餐厅推荐、交通优化、景点推荐细分等能力。
- **一致性**：前后端使用统一 DTO / Schema。
- **可追踪性**：保留版本历史、任务状态与操作记录。
- **可靠性**：Agent 输出需经过严格 schema 校验，失败时支持重试与降级。
- **性能**：生成任务采用异步化处理，避免长请求阻塞。

## 4. 总体架构设计

### 4.1 分层架构
系统采用前后端分离 + AI 编排层的方式，整体分为五层：

1. **表现层（Frontend）**：React 页面，负责表单、展示、编辑和历史方案浏览。
2. **接口层（API）**：FastAPI REST 接口，完成鉴权、DTO 校验和业务路由。
3. **编排层（LangGraph）**：将多个 Agent 组织成流水线状态图，控制旅行规划与画像更新。
4. **数据层（MySQL / Redis）**：保存用户、画像、方案、版本、任务状态与日志。
5. **外部能力层**：天气、地图、POI、图片、LLM 等第三方服务。

### 4.2 核心数据流
1. 用户登录后进入前端。
2. 用户创建旅行需求。
3. FastAPI 接收请求并校验。
4. 系统创建异步任务，返回 task_id。
5. LangGraph 根据流水线依次调用各个 Agent。
6. Agent 调用外部服务，生成中间结果。
7. Planner Agent 组装最终方案。
8. 方案落库到 MySQL（半结构化存储）。
9. 前端通过任务状态接口轮询结果并展示。
10. 用户打开方案时系统再次检查天气预警并展示风险提示。
11. 用户基于历史版本再次发起生成，形成版本链。

### 4.3 架构特点
- 使用 LangGraph 实现受控流水线，而不是复杂并行协作。
- 使用 MySQL + JSON 半结构化存储平衡查询能力与灵活性。
- 使用异步任务机制处理长耗时生成流程。
- 用户画像由表单输入 + 历史行为补充，形成可持续改进的个性化能力。
- 天气预警不仅用于提示，也可作为规划约束输入。

## 5. 领域模型与数据设计

### 5.1 核心实体
- **User**：用户账号与身份信息。
- **UserProfile**：用户画像。
- **TripPlan**：旅行方案逻辑对象。
- **TripPlanVersion**：方案版本。
- **PlanTask**：方案生成任务状态。

### 5.2 方案版本模型
本项目采用“版本链”而非多人协作模型：
- 原始方案创建后可生成多个版本。
- 用户可以基于历史版本继续生成新版本。
- 所有修改均产生新版本，不覆盖旧版本。
- 保留每个版本的创建来源、时间和内容快照。

### 5.3 用户画像模型
画像字段：
- travel_style：休闲 / 深度游 / 特种兵 / 商务等
- budget_level：低 / 中 / 高
- interest_tags：历史、自然、美食、亲子、购物等
- transport_preference：步行 / 公交 / 打车 / 自驾
- accommodation_preference：经济型 / 舒适型 / 高端型
- risk_sensitivity：对天气变化的敏感程度
- pace_preference：松散 / 中等 / 紧凑

### 5.4 旅行方案 JSON（半结构化）
将 TripPlan 的完整方案内容存入 JSON 字段，内容包括：
- 基本信息：city、start_date、end_date、days
- 每日安排：days[]
- 景点列表：attractions
- 酒店信息：hotel
- 餐饮信息：meals
- 天气信息：weather_info
- 预算信息：budget
- 预警信息：warnings
- 总体建议：overall_suggestions

同时，将常用查询字段拆出为普通列，例如 city、start_date、end_date、budget_range、status 等，便于检索。

## 8. 流水线式 LangGraph 设计

### 8.1 设计目标
LangGraph 的职责是将多个 Agent 组织成一条可控的流水线，避免单 Agent 承担所有任务导致提示词过长、工具调用混乱和调试困难。当前版本不做复杂并行协作，而是采用顺序执行与节点校验的方式。

### 8.2 Graph State 建议
统一状态对象建议包含：
- user_id
- request
- user_profile
- attractions
- weather
- hotels
- draft_plan
- warnings
- final_plan
- version_info
- task_info
- error_info

### 8.3 Agent 划分

#### 8.3.1 UserProfile Agent
职责：
- 读取用户表单信息与历史方案行为。
- 生成用户画像摘要。
- 更新用户画像标签。
- 为后续规划提供画像上下文。

#### 8.3.2 Attraction Agent
职责：
- 根据城市与偏好搜索景点。
- 返回结构化景点候选列表。

#### 8.3.3 Weather Agent
职责：
- 查询旅行日期范围内天气信息。
- 提供天气趋势与风险点。
- 生成天气风险评分，为 Planner 提供约束输入。

#### 8.3.4 Hotel Agent
职责：
- 根据预算和住宿偏好搜索酒店。
- 返回候选酒店与估算费用。

#### 8.3.5 Planner Agent
职责：
- 整合景点、天气、酒店和用户画像。
- 生成完整行程 JSON。
- 生成预算、每日安排、总体建议与风险提示。

#### 8.3.6 Alert Check Node
职责：
- 在方案生成后或方案详情打开时检查天气预警。
- 输出预警项和提示文案。
- 不直接修改历史版本内容。

### 8.4 Graph 流程
推荐流程如下：
1. 输入请求。
2. 构建或更新用户画像。
3. 顺序获取景点候选、天气信息、酒店候选。
4. 对每一步结果进行 schema 校验。
5. Planner Agent 生成草案。
6. Alert Check Node 计算预警并附加风险信息。
7. 输出最终方案。
8. 保存方案版本。
9. 用户再次发起生成时，根据历史版本作为输入之一继续生成新版本。

### 8.5 Agent 失败处理
- JSON 解析失败：重试一次或两次。
- 外部服务失败：降级处理，不阻断整体生成。
- 某节点返回空数据：使用默认模板补位。
- 关键字段缺失：拒绝入库并提示用户重试。
- 超时：任务标记失败并返回错误信息。

## 9. 天气预警机制设计

### 9.1 触发时机
- 行程创建或重新生成时：将天气风险作为约束输入。
- 行程详情页打开时：再次检查并展示最新天气预警。

### 9.2 预警规则
根据天气服务返回结果判断以下风险：
- 暴雨 / 大雨
- 高温 / 极端低温
- 大风
- 雷暴
- 台风或其他强预警

### 9.3 风险表达方式
在 TripPlan 的 `warnings` 中存储如下内容：

```json
{
  "date": "2026-05-01",
  "type": "heavy_rain",
  "level": "high",
  "impact": "户外景点受影响",
  "suggestion": "建议将该日行程调整为室内活动"
}
```

### 9.4 处理方式
- 前端展示预警卡片。
- 给出调整建议：改为室内景点、缩短户外停留时间、调整出行顺序。
- 方案生成阶段可使用 `weather_risk_score` 参与排序与筛选。
- 打开历史方案时，只提示风险，不强制改写历史记录。

### 9.5 风险评分示例
- 晴天：+2
- 多云：+1
- 小雨：-1
- 中雨：-3
- 暴雨：-5
- 大风 / 雷暴 / 台风：-4 ~ -6

## 10. 历史方案与版本机制设计

### 10.1 设计思路
系统只保留单用户个人历史方案。每次重新生成、编辑保存都产生新版本，形成版本链。

### 10.2 版本链规则
- 创建方案产生 v1。
- 基于某历史版本继续生成时，产生新版本。
- 新版本保留 `parent_version_id`，用于追踪来源。
- 当前版本由 `trip_plans.current_version_id` 指向最新版本。
- 历史版本可查看，但默认不可覆盖。

### 10.3 查询能力
用户可以：
- 查看历史方案列表。
- 查看单个方案的版本历史。
- 查看某版本详情。
- 基于历史版本再次发起生成任务。

## 11. 认证与授权设计

### 11.1 登录方式
- 使用 JWT 进行身份认证。
- 课程项目可采用简化方案，仅使用 access_token。

### 11.2 登录流程
1. 用户登录成功后返回 JWT。
2. 前端保存 token。
3. 后续请求携带 Authorization header。
4. FastAPI 中间件解析 token 并注入用户身份。

### 11.3 权限控制
- 未登录用户：不能创建、保存或查看个人历史方案。
- 登录用户：可创建方案、查看自己的历史方案和版本历史。
- 所有历史记录默认仅对本人可见。

## 13. 异步任务设计

### 13.1 设计目标
行程生成涉及 LLM 推理、外部天气查询、POI 查询与版本落库，属于耗时操作。因此需要采用异步任务方式处理，避免前端阻塞和接口超时。

### 13.2 任务流程
1. 前端提交创建行程请求。
2. 后端立即创建 `plan_tasks` 记录。
3. 后端返回 `task_id`。
4. 后台 worker 执行 LangGraph 流水线。
5. 每个阶段更新任务状态。
6. 任务完成后写入方案版本并更新 `result_version_id`。
7. 前端轮询任务状态直至成功或失败。

### 13.3 任务状态
- `pending`：已创建，等待执行。
- `running`：执行中。
- `success`：执行成功。
- `failed`：执行失败。

### 13.4 实现建议
- 课程项目优先使用 FastAPI BackgroundTasks 或轻量队列方案。
- 若需要更完整的生产式设计，可采用 Celery + Redis。
- 前端可通过轮询实现状态更新，必要时可扩展 SSE。

### 13.5 超时与失败
- 任务超时后置为 failed。
- 记录 error_message，方便排查。
- 前端展示失败原因，并允许用户重新提交请求。

## 16. 关键实现策略

### 16.1 schema 严格校验
- Agent 输出必须先映射到 Pydantic 模型。
- 失败时记录错误并重试。
- 通过后才能写入数据库。

### 16.2 半结构化存储策略
- TripPlan 完整方案存 JSON。
- 常用筛选字段单独落表。
- 版本管理存元数据表。
- 便于 schema 演进与数据检索。

### 16.3 版本生成策略
- 创建方案产生 v1。
- 编辑方案产生新版本。
- 基于历史版本重新生成产生新版本。
- 所有版本保留历史。

### 16.4 容错策略
- 景点失败：返回空列表或默认推荐。
- 酒店失败：返回基础占位方案。
- 天气失败：仅跳过预警，不阻断主流程。
- LLM 失败：重试后给出失败提示。
- 任务异常：标记失败并写入错误信息。

### 16.5 可观测性
- 记录 task_id、agent 输入输出、错误栈与执行时间。
- 便于调试异步任务和多阶段编排。
